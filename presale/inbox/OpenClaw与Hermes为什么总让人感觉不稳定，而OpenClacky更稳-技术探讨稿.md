# Agent 不是不能用，而是 gateway 太容易成为故障中心：聊聊 OpenClaw、Hermes 与 OpenClacky

这段时间，OpenClaw 和 Hermes 都很火。

前者代表了一种“本地 AI Agent 操作系统”的想象：技能市场、跨平台 IM 接入、本地执行、自动化编排，生态很强，讨论度也很高。后者则把“自进化”“记忆系统”“越用越强”这些概念推到了台前，给很多技术用户带来了新的期待。

但如果你真的去群里潜水、看用户反馈、自己跑过几轮配置，很容易发现一个共同体感：**它们都很强，但也都很容易不稳定。**

OpenClaw 用户会遇到双通道一开就卡死、Telegram 发图直接把 gateway 打崩、消息半天不回复、版本一升级问题变多、配置来回检查还是跑不通。Hermes 用户的典型反馈则更直接：**“这一卡就完了。”** 再加上 API 兼容性、接入文档不完整、配置链路依赖 workaround，整体使用门槛并不低。

很多人会把这类现象理解成一句话：**这些产品还早期，所以 bug 多。**

这当然没错，但如果只把问题归结为“bug 多”，其实还不够。**从工程角度看，这背后往往不是单点 bug，而是架构代价。**

## 不稳定，不只是因为 bug，而是因为 gateway 天然会变复杂

在现有资料里，OpenClaw 的典型工作流里有一个非常关键的环节：

**用户发送指令 → OpenClaw 网关处理认证 → AI 模型理解意图 → 调用 Skill/Tool → 执行结果返回。**

这里的“网关”并不是一个可有可无的小组件，它实际上承担了大量中枢职责：

- 认证与鉴权
- 消息接入与路由
- 会话状态管理
- 多通道连接维护
- 错误恢复与重连
- 某些场景下的配置同步与转发

一旦系统把这些职责堆到 gateway 上，问题就来了：**它会天然成为复杂性中心，也很容易变成故障中心。**

为什么？因为 AI Agent 这类系统本身就已经够复杂了。上面有模型推理和上下文管理，下面有本地文件、浏览器、命令行、外部 API，再往外还有微信、飞书、Telegram 等不同通道。你再在中间放一个需要长期维持状态的 gateway，相当于又加了一层高耦合系统。

这时候，任何一个环节的波动，都可能被放大：

- 上游通道消息格式变化，gateway 要适配
- 网络短时抖动，连接状态可能错乱
- 会话缓存积累过多，路由或响应开始变慢
- 认证逻辑、设备绑定、端口暴露等带来额外风险
- 某条通道的特殊消息类型，比如图片、群聊、双通道切换，容易触发边界问题

所以用户体感上的“卡”“不回”“重启才好”“升级后炸了”，往往不是某一个函数写错这么简单，而是**中间那一层承担了太多职责，任何一处出问题都会影响整体稳定性。**

## 为什么 OpenClaw 和 Hermes 容易出现“看起来随机”的故障

如果只看舆情素材，会发现很多故障表面上彼此无关，但它们其实有共同结构。

OpenClaw 这边，已有反馈包括：

- 开启微信双通道后直接卡死
- 通过 Telegram 发图片，gateway 崩溃，只能手动重启
- `MEMORY.md` 内容一多，就开始巨卡，群里不回复，私聊也容易崩
- 新版本迭代很快，但也意味着问题频繁波动
- 部分安装、升级、SkillHub 显示、403 权限等问题，往往都带着明显的“链路脆弱感”

Hermes 这边则是另一种早期架构的典型问题：

- 一旦遇到网络抖动或 API 超时，缺少自恢复余量
- API 兼容性不够稳，需要手动改 endpoint 或配置兼容层
- 文档不足，接入路径不清晰，很多配置只能靠群友互相救火
- 调教成本高，系统不是“装上就用”，而是“装上之后还要继续养”

这些问题看起来分散，但都指向一个共同事实：**系统越依赖中枢层去统筹所有状态，越容易出现“局部波动拖垮整体体验”的现象。**

这也是为什么很多用户会觉得这类产品“玄学”。

不是因为它们没有工程能力，而是因为当认证、连接、消息路由、状态同步、会话上下文、外部通道差异都压在一条链上时，系统故障不一定表现为明确报错，反而更常表现为：

- 偶发卡住
- 响应变慢
- 某个通道突然失灵
- 重启后恢复，过一阵又出问题
- 同样的配置昨天能跑，今天又不行

这类问题，对终端用户来说最难受。因为它不像编译失败那样直接给你一条错误信息，而是让你不停怀疑：到底是模型问题、配置问题、网络问题、权限问题，还是 gateway 又抽风了？

## gateway 架构不是错，只是它的代价常被低估

这里要说一句公道话：**gateway 架构本身不是原罪。**

它有它的优势。比如统一接入多通道、集中做认证、方便扩展生态、适合后续加管理能力、监控能力和平台化能力。从平台产品的角度看，这条路线并不奇怪，甚至很自然。

问题在于，很多人先感受到的是“扩展性想象”，但最后要天天面对的却是“运行时复杂性”。

也就是说，gateway 架构的收益主要体现在：

- 功能整合能力
- 平台扩展能力
- 多通道统一管理
- 生态式能力拼装

而它的代价则体现在：

- 运行链路更长
- 排障边界更多
- 状态一致性更难维护
- 出错点更分散
- 对部署环境、网络环境、配置质量更敏感

如果一个用户的核心诉求是“玩最强生态、试最多能力、追最新框架”，那这个代价未必不能接受。

但如果一个用户真正想要的是：**把 Agent 放进日常工作流，稳定地跑起来，今天能用，明天也能用，别动不动去修中间层。** 那么架构选择就会变得很重要。

## OpenClacky 为什么更稳：不是更重，而是更轻

OpenClacky 在这件事上的思路，恰恰是反过来的。

在现有内部材料里，对 OpenClacky 的产品力有一句非常关键的总结：

**“去网关化，进程起停简单，稳定不掉线（对比爱马仕/Hermes 网关式架构）。”**

这句话其实已经把核心差异讲透了。

OpenClacky 没有把系统设计成一个高度依赖 gateway 中枢持续维持状态的架构，而是更偏向轻量、本地进程式、去网关化的实现思路。这样做带来的直接好处，不一定是“功能看起来更炫”，但往往是：

- 链路更短，少一层中枢转发
- 进程职责更清晰，问题更容易定位
- 起停更简单，异常恢复成本更低
- 少一层长期连接状态管理，掉线风险更小
- 对本地用户来说，部署和卸载都更干净

从工程上讲，**少一层，不只是少一个组件，而是少一整类问题。**

少掉的是哪些问题？

- gateway 本身的连接维护问题
- 中枢状态同步问题
- 路由与转发链路问题
- 某些端口暴露和认证面的问题
- 通道兼容差异在中间层被放大的问题

这不代表 OpenClacky 就没有 bug，也不代表轻架构就天然完美。任何实际可用的软件，只要接入模型、通道、技能系统，就一定会遇到边界情况。

但区别在于：**当系统不把过多职责压在 gateway 上时，很多故障不会被放大成“整条链都不工作”。** 这对实际使用体验的影响非常大。

## 如果目标是真正用起来，OpenClacky 的路线更务实

过去一年，很多 AI Agent 产品都在追求“平台感”“生态感”“全栈控制感”。这很有吸引力，也很容易讲故事。

但从用户真正长期使用的角度看，最重要的问题往往不是“能不能接 10 个通道”，而是：

- 你今天发出去的消息，它回不回？
- 你让它执行工作流，它会不会半路挂掉？
- 你升级一次版本，要不要再花半天排障？
- 你遇到异常时，能不能快速恢复，而不是重装一轮？

这也是为什么我们越来越觉得，OpenClacky 这条路线更务实。

它不是试图把所有复杂性都集中管理，而是通过去网关化、轻进程化的思路，把系统收缩到更容易稳定运行的范围内。对很多技术用户来说，这不是“保守”，而是成熟。

因为真正阻碍 Agent 进入日常工作的，从来不只是模型能力不够强，更多时候是**运行稳定性不够可靠**。

当你不是在 demo，而是在真实工作里使用它，稳定就不再是一个附加项，而是第一优先级。

## 最后：什么人适合哪条路线？

如果你追求的是更强的生态整合、更复杂的多通道平台化能力，也愿意接受更高的配置复杂度和更重的维护成本，那么 OpenClaw / Hermes 这类路线依然有它们的吸引力。

但如果你更在意的是：

- 安装和运行简单
- 不想天天排查 gateway
- 希望 Agent 真正稳定进入自己的工作流
- 希望问题出现时能快速恢复，而不是全链路怀疑人生

那么 OpenClacky 这种**去网关化、轻本地进程**的路线，确实更值得认真看一眼。

说到底，这不是“谁更先进”的问题，而是**你到底想养一个需要持续调教的系统，还是想用一个能长期稳定工作的工具。**
