## 一、OpenClaw 技术架构解析

### 1.1 核心定位

OpenClaw 是一个**本地优先的 AI Agent 编排框架**，被称为"个人 AI 的操作系统"。它的核心创新在于将大模型的智能与具体能力**解耦**，通过标准化的接口和技能市场（ClawHub），让模型无需关心能力实现，能力也无需为每个模型适配。

**GitHub 数据**（截至 2026-03）：
- 约 26 万 Stars，超越 React（24.3万）和 Linux（21.8万），成为 GitHub 史上 Star 最多的项目
- 4 个月从零登顶，全球增长最快的开源 AI Agent 项目

---

### 1.2 核心架构设计

#### 三层架构

```
┌─────────────────────────────────────────────┐
│         应用层：IM 驱动（微信/飞书/Slack...）    │
├─────────────────────────────────────────────┤
│       框架层：智能体编排 + 权限管理 + Skills     │
├─────────────────────────────────────────────┤
│        执行层：本地执行（文件/终端/API/浏览器）   │
└─────────────────────────────────────────────┘
```

#### 本地优先架构
- **所有代码、数据、记忆都在本地运行**，不依赖云端 API
- AI 能够直接执行操作系统级操作：读写文件、执行 Shell 命令、调用私有 API
- 这是 OpenClaw 与其他云端 AI 助手的根本差异

#### 权限系统（设计亮点 + 争议焦点）
- 每个 Skill 需要**明确声明所需权限**（读取文件、写入文件、执行命令、访问网络等）
- 用户安装 Skill 时看到清晰的权限列表，像手机 App 安装时的权限请求
- 但批评者指出：默认以**启动用户完整系统权限**运行，与"最小权限"原则背道而驰

#### 技能市场（ClawHub）
- 插件化的 Skills 机制，开发者可以为 AI 赋予任意新能力
- 社区 Skills 三个月内被超过 5 万开发者使用
- **安全隐患**：ClawHub 已发现大量恶意技能包，供应链投毒问题严重

---

### 1.3 核心工作流程

```
用户发送指令 → OpenClaw 网关处理认证
    → AI 模型理解意图 + 读取本地记忆（MEMORY.md）
    → 决策调用哪个 Skill/Tool
    → Skill 执行操作（可调用外部资源）
    → 结果写回记忆系统 → 响应用户
```

**关键特性**：
- 支持 MCP（Model Context Protocol）协议
- 基于 Claude Code（Anthropic Agents SDK）执行
- 具备"涌现能力"：遇到未预编程的任务时，会自主调用系统工具链解决

## 二、Hermes Agent 技术架构解析

### 2.1 核心定位

Hermes Agent 是 **Nous Research**（2026 年 2 月）发布的开源 AI Agent 框架，核心卖点是**自进化能力**——Agent 运行产生的轨迹数据，通过自研的 **Atropos** 闭环反哺到下一代 Hermes 模型，形成"越用越强"的飞轮。

GitHub 约 3 万 Stars（两个月），被称为 OpenClaw"第一个真正意义上的竞争对手"。

---

### 2.2 核心架构：四层内存设计 + 自进化循环

```
┌────────────────────────────────────────────┐
│           应用层：多场景定制与扩展            │
├────────────────────────────────────────────┤
│       ReAct Loop（双重迭代保护 + 工具调用）    │
├────────────────────────────────────────────┤
│        四层内存：短/中/长/永久              │
├────────────────────────────────────────────┤
│     Atropos 闭环：运行轨迹 → 模型更新        │
└────────────────────────────────────────────┘
```

#### 自进化原理
- Agent 运行产生"解决问题的轨迹数据"
- Atropos 机制将这些轨迹数据**反哺训练下一代模型**
- 本质：把"训练好的模型"和"跑模型的 Agent 框架"两端打通

#### 四层内存设计
- **短时记忆**：单次会话内的上下文
- **中期记忆**：跨会话的当前任务状态
- **长时记忆**：积累的经验和技能
- **永久记忆**：知识沉淀，可跨任务复用

**核心创新**：真正的 AI Agent 不只是执行任务的工具，而是能够通过 **Skill 沉淀**实现知识的积累和传承

## 三、Nanobot vs NanoClaw：OpenClaw 生态分支

### 3.1 Nanobot（港大 HKUDS，22k Stars）

**定位**：大道至简，Python 原生

**核心特点**：
- 仅用 **4000 行核心代码**实现了 Agent Loop、上下文构建、记忆管理
- 代码可读性极强，资源占用低
- 基于 **MCP 协议**与插件化技能包
- 支持 Markdown 格式的"人类可读"记忆系统
- 由 Claude Code 引导设置，完全"AI 原生"安装体验

**优势**：极简、Python 原生、MCP 支持 → 精准击中 AI 开发者痛点
**劣势**：没有传统安装向导，前置依赖多（Python + Claude Code）

---

### 3.2 NanoClaw（9.5k Stars）

**定位**：安全容器版 OpenClaw，"防弹手机"

**核心特点**：
- **容器级隔离**——每个 AI 操作都在独立 Linux 容器中执行
- 解决了 OpenClaw"权限过大"的安全原罪
- 但功能相对精简，安全性拉满但能力受限
- 深度绑定 Anthropic 生态，模型选择最受限
- 前置依赖：Node.js + Claude Code + Docker

**优势**：系统级安全隔离，适合高安全场景
**劣势**：生态最受限，技能包数量远少于 OpenClaw

## 四、OpenClaw 安全问题：深度整理

> 这是竞品分析中对我们最有价值的攻击面

### 4.1 已披露的安全事件

#### ① CVE-2026-25593：WebSocket 网关密码暴力破解
- **漏洞原理**：OpenClaw 在本地运行一个 WebSocket 网关（默认端口 18789），默认绑定到 `0.0.0.0:18789`（非仅本地）。设计者错误假设"来自本地的流量可信"，对 localhost 连接**豁免暴力破解限流 + 自动批准新设备配对 + 无日志**。现代浏览器不阻止网页向 localhost 发 WebSocket 连接。
- **攻击链**：用户访问恶意网站 → JS 后台连接本地 18789 → 每秒数百次暴力破解密码 → 几秒~几分钟攻破 → 自动注册为受信任设备 → 完全接管
- **影响**：窃取所有配置（API 密钥、Slack/GitHub 令牌、聊天记录）、执行任意 Shell 命令、读取本地文件
- **严重程度**：**CVE-2026-25593，CVSS 评分预计 9.8/10**

#### ② CVE-2026-32302：特权管理会话绕过
- 允许攻击者通过特定配置绕过来源验证，获取特权管理会话

#### ④ 全球公网暴露规模

### 4.5 安全问题的结构性原因

#### 提示注入（Prompt Injection）——天生缺陷
- **原理**：LLM 看到的是连续的 Token 序列，无法在语义层面区分"指令"和"数据"。攻击者把恶意指令藏在网页、邮件、文档中，AI 读取时指令被执行。
- **为什么无法根治**：SQL 注入可用预编译语句防御（区分命令 vs 数据），但 LLM 的"指令"和"数据"都是 Token，架构层面就融合在一起。
- **官方建议**：使用更强模型（如 Opus）。官方实际上**默认承认这个问题无解**，只是"更强的模型更难被劫持"。

#### 幻觉（Hallucination）——LLM 基因缺陷
- 幻觉是 LLM 泛化能力的副作用，无法剥离
- 失控场景：AI 可能在特定条件下产生不可预测行为
- Summer Yue 事件部分归因于此
