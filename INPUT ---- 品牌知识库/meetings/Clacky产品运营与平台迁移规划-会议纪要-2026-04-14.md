## 二、重大决策：Clacky平台 → OpenClacky 迁移

**背景问题**：
- railway权限不完整（project级TOKEN有bug），导致部署500、AI无法拿log等连锁问题
- PG数据库部署被迫在服务端绕路，维护成本极高
- 云端CDE体系（Paas）太重，自研成本过大

**决策结论**：
- 将Clacky平台上的LM等核心能力迁移回OpenClacky
- 用Railway原生模板替代原有封装体系
- 预计1-2天完成技术迁移
- 用户需自行注册Railway账户
- Clacky平台后台功能关闭，重定向至OpenClacky

**待处理**：已有部署用户妥善安置

## 三、产品力提炼（竞争定位）

| 维度 | Clacky/OpenClacky 优势 |
|------|----------------------|
| 国产化 | 访问Claude无墙，支持Windows/Mac/WSL |
| 安装 | 一键安装包，适合国内网络环境 |
| 架构 | 去网关化，进程起停简单，稳定不掉线（对比爱马仕/Hermes网关式架构） |
| 成本 | 每条线打成本标签，成本控制中心（营销功能） |
| 变现 | "养得起"概念，省钱+可变现，可导入其他平台Skill |
| 安全 | 本地进程，卸载干净，无后顾之忧 |
| 定位 | 对标Claude Code，"养虾"概念比"爱马仕"门槛低10倍 |
| 响应 | 与创作者统一战线，迭代速度是去中心化产品不具备的优势 |

**对标竞品**：
- 爱马仕(Hermes)：记忆系统/人设背书强，但安装不易、架构重
- OpenClaw：去中心化，反馈慢
- Claude Code：贵但强
